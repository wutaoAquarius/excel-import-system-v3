# 万能导入 V2 —— 需求文档

## 一、项目定位

物流/快递行业批量下单 Web 应用。客户上传任意格式文件（Excel / Word / PDF），系统通过**规则引擎 + LLM 辅助**将其解析为结构化运单数据，完成批量下单并持久化存储。

---

## 二、技术栈约束

| 项目 | 要求 |
|---|---|
| 框架 | Next.js App Router + TypeScript |
| 部署 | Vercel（必须，无 URL 则 0 分） |
| UI | 主色 `#0fc6c2`，圆角卡片，鲸天系统风格 |
| LLM | DeepSeek / GPT / Claude 等，API 调用 |
| 数据库 | Neon / Supabase / Turso（通过 Vercel Marketplace 集成） |

---

## 三、核心数据模型：运单字段

每条**出库单**按**外部编码**聚合，同一外部编码下多个 SKU 行共享一组收货信息。

| 字段 | 必填规则 |
|---|---|
| 外部编码 | 否（用于去重/聚合） |
| 收货门店 | A组/B组二选一 |
| 收件人姓名 | A组/B组二选一 |
| 收件人电话 | A组/B组二选一 |
| 收件人地址 | A组/B组二选一 |
| SKU物品编码 | **必填** |
| SKU物品名称 | **必填** |
| SKU发货数量 | **必填**，正数 |
| SKU规格型号 | 否 |
| 备注 | 否 |

**A组（门店模式）**：只填"收货门店"即可
**B组（收件人模式）**：需填"姓名 + 电话 + 地址"三个字段
两组都没填则校验不通过。

## 四、功能模块详细需求

### 模块一：解析规则配置 + AI 辅助生成（核心，50分）

**设计原则**：不得为任何一份文件硬编码解析逻辑，代码中不得出现文件名判断或特定列名硬编码。必须设计一套**通用规则描述语言/规则引擎**，新增格式只需配置规则，代码零改动。

**规则管理**：
- 规则持久化存储到服务端数据库
- 支持创建、编辑、删除、复制
- 提供"新建规则"按钮，用户手动选择使用哪条规则（不做自动匹配）

**AI 辅助生成规则（关键能力）**：
- 上传新文件后，LLM 分析文件结构，生成**推荐规则**（不是直接解析数据，而是生成规则配置）
- AI 需标注哪些映射是推测的
- 用户在推荐规则基础上**手动微调确认**后保存，不直接生效

**规则预览测试**：
- 保存前可用当前文件试解析，实时预览结果，确认正确后再保存

**规则引擎需覆盖的复杂场景**：

| 场景类型 | 说明 |
|---|---|
| 头部跳过 | 跳过文件前 N 行的干扰信息 |
| 尾部信息提取 | 收货人/电话/地址散落在数据区之外的独立行 |
| 跨行聚合 | 按配送单号聚合，多 SKU 行共享一组收货人信息 |
| 矩阵转置 | 门店/日期作为列头横向排列，需转置为独立运单记录 |
| 多Sheet合并 | 遍历所有 Sheet，每个 Sheet 独立解析后合并 |
| 卡片式拆分 | 按卡片边界标志识别，每个卡片内有独立小表和收货信息 |
| 纯文本解析 | 无表格的 Word 文档，用正则从段落文本提取字段 |
| 复合单元格拆分 | 单个单元格内含"物品名x数量\n物品名x数量"，需拆为多行 |
| PDF多单拆分 | 同一 PDF 内含多个独立订单，识别分隔线/页面边界后拆分配对 |

**需兼容的 9 种出库单**（每少兼容 1 份扣 3 分，不足 5 份本项 0 分）：

| 文件 | 格式 | 核心难点 |
|---|---|---|
| 黎明屯配送发货单 | Excel | 尾部信息区提取收货人（横向排列在数据区外） |
| 湖南仓发货明细 | Excel | 跨行聚合（按配送单号分组） |
| 欢乐牧场模板 | Excel | 矩阵转置（SKU×门店矩阵） |
| 黔寨寨配送单 | PDF | PDF解析 + 底部纯文本区收货人 |
| 多门店分Sheet出库单 | Excel | 多Sheet合并 + 每Sheet底部横向收货人 |
| 门店调拨单（卡片式） | Excel | 卡片边界识别（"▶ 调拨记录 #N"为起始标志） |
| 门店配送确认单 | Word | 纯文本解析（"━━━"分隔线划分记录边界） |
| 周配送计划 | Excel | 双重转置 + 复合单元格拆分 |
| 配送签收单（多单PDF） | PDF | PDF多订单拆分（一个PDF含3个独立签收单） |

### 模块二：文件导入与解析执行

**支持格式**：Excel（.xlsx/.xls）、Word（.docx）、PDF
**上传方式**：拖拽 + 点击

**解析流程**：
```
上传文件
  → 用户手动选择已有规则 或 点击"新建规则"
    → 若新建：LLM 预处理分析文件 → 生成推荐规则 → 用户微调确认
  → 执行解析
  → 展示结果
```

**其他要求**：
- 导入时显示实时进度条（百分比 + 当前处理条数/总条数）
- 完善错误处理：格式错误、文件为空、编码异常、无法解析均有明确提示
- 解析失败时展示原始文件信息，并提供手动配置规则的入口

---

### 模块三：数据预览与编辑

- 类 Excel 表格形式展示，表头固定，支持横向滚动
- 单元格点击可直接编辑（Tab/回车切换，类在线 Excel 体验）
- **行内错误实时校验**：必填字段缺失标红、电话格式错误标红、数量非正数标红
- **全部错误一次性展示**：同时列出所有错误，标注行数 + 字段名 + 错误原因
- **外部编码重复检测**：同批次内重复 + 与数据库已有数据重复，均高亮提示
- 支持删除行、新增空行
- 支持将预览数据（含修改）导出为 Excel 文件

---

### 模块四：提交下单

- 有错误行不允许提交，提示用户先修正
- 点击"提交下单"显示上传进度条
- 提交成功后数据持久化到数据库
- 返回提交结果汇总：成功 N 条、失败 N 条

---

### 模块五：已导入运单列表

- 查看所有历史导入记录（从数据库读取）
- 支持按外部编码、收件人姓名、提交时间筛选/搜索
- 支持分页

## 七、关键设计约束（不可违反）

1. **禁止硬编码**：代码中不能出现文件名判断、特定列名、特定行号的硬编码逻辑
2. **规则驱动**：所有格式适配通过配置规则实现，代码层面对所有文件一视同仁
3. **AI 生成规则而非数据**：LLM 的职责是分析文件结构并输出规则配置，而不是直接返回解析后的数据
4. **用户确认**：AI 生成的规则必须经过用户确认（可微调）后才能生效
5. **API Key 安全**：LLM 调用必须在服务端，Key 不能暴露到前端
